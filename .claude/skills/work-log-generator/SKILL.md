# 업무일지 생성기

## 워크플로우 (토큰 절약 우선)
1. 사용자가 날짜를 지정하면 해당 날짜의 커밋을 조회, 미지정 시 오늘 날짜 사용
2. `git log --after="YYYY-MM-DD 00:00" --before="YYYY-MM-DD 23:59" --format="%h %s%n%b" --stat` 로 커밋 메시지 + 변경 파일 목록을 **한 번에** 조회
3. 커밋 메시지와 --stat 만으로 업무일지 작성 (대부분의 경우 이것만으로 충분)
4. 커밋 메시지가 불명확한 경우에만 `git diff <hash>^..<hash> -- <특정파일> | head -80` 으로 **핵심 파일만 제한적으로** 확인
5. [추가]와 [수정] 섹션으로 분류
6. 아래 출력 형식에 맞춰 업무일지 작성

## 토큰 절약 규칙 (반드시 준수)
- git diff 전체를 읽지 말 것. --stat + 커밋 메시지 우선 활용
- diff 확인이 필요한 경우 `head -80` 등으로 출력량 제한
- .sql, .css, gwdb.sql 등 대용량/생성 파일은 diff 조회하지 말 것
- 여러 커밋의 stat은 하나의 git log 명령으로 한 번에 조회

## 분류 기준
- **[추가]**: 새로운 파일 생성, 신규 기능 개발, 새로운 API/모델/뷰 추가
- **[수정]**: 기존 코드 수정, 버그 수정, 리팩토링, 로직 변경, 코드 정리, 보안 패치

## 출력 형식 (반드시 이 형식 준수)

```
**YYYY-MM-DD (요일) 업무일지**

**[ 추가 ]**

▷ 대분류 제목 (기능 단위로 묶기)
- 소분류 상세 설명 (일반인도 이해할 수 있는 수준)
- 소분류 상세 설명
- ...

▷ 대분류 제목
- 소분류 상세 설명
- ...

**[ 수정 ]**

▷ 대분류 제목
- 소분류 상세 설명
- ...
```

## 작성 규칙
- 대분류(▷): 관련 기능을 하나의 주제로 묶어 제목 작성 (예: "출퇴근 관리 API 신규 구축", "XSS 보안 취약점 수정")
- 소분류(-): 구체적인 변경사항을 일반인이 이해할 수 있도록 서술
- 각 섹션([추가]/[수정]) 내 소분류 합계가 **6줄 이상 10줄 이하** 되도록 작성
- 추가/수정 중 해당사항이 없으면 해당 섹션 생략 가능
- 변수명, 함수명 등 기술 용어는 괄호 안에 보충 설명 추가
- 단순 주석 정리, 공백 변경 등 사소한 변경은 생략하거나 "코드 정리" 항목으로 통합
- 커밋이 여러 개일 경우 내용을 통합하여 하나의 업무일지로 작성
- 요일은 한글로 표기 (월, 화, 수, 목, 금, 토, 일)
